## 1 项目背景与需求分析

### 1.1 项目背景

高校校园设施维修管理长期依赖人工电话报修或线下填单，存在信息传递滞后、任务分配不透明、进度追踪缺失等问题。学生报修后无法实时了解维修进展，维修部门缺乏有效的工单调度与绩效统计手段，导致维修响应慢、学生满意度低。

本项目"校园智能报修服务平台"以高校后勤维修管理场景为背景，构建一套覆盖报修提交、工单派发、维修执行、服务评价的全流程数字化管理系统。系统采用前后端分离架构，通过 Web 浏览器提供统一入口，支持学生、维修工、管理员三类角色协同工作，旨在提升维修响应效率和服务透明度。

### 1.2 涉众分析

| 涉众 | 角色描述 | 主要职责 | 关注点与期望 |
|------|---------|---------|-------------|
| 学生 | 报修发起方 | 提交报修、查看进度、评价服务 | 操作简便、进度实时可见、问题快速解决 |
| 维修工 | 维修执行方 | 接单、执行维修、填写结果 | 任务清晰、优先级明确、工作记录规范 |
| 管理员 | 平台运营方 | 派单调度、用户管理、公告发布、数据统计 | 全局可视、调度灵活、数据准确 |
| 后勤部门 | 组织管理方 | 监督维修质量、绩效考核 | 统计报表完整、工单流程合规 |

### 2.2 领域模型

核心领域对象及关系：

```
用户(User)
  ├── id: Long
  ├── username: String (学工号)
  ├── realName: String
  ├── role: Integer (0学生/1维修工/2管理员)
  └── status: Integer (0禁用/1启用)

报修工单(RepairOrder)
  ├── id: Long
  ├── orderNo: String (唯一编号)
  ├── studentId → User
  ├── workerId → User
  ├── category: Integer (1水电/2网络/3门窗/4电器/5其他)
  ├── location: String
  ├── description: String
  ├── status: Integer (0待派单/1已派单/2处理中/3已完成/4已评价/5已取消)
  ├── priority: Integer (1普通/2紧急/3非常紧急)
  └── 时间戳: assignTime/acceptTime/finishTime/createTime

评价(Evaluation)
  ├── orderId → RepairOrder (一对一)
  ├── studentId → User
  ├── workerId → User
  ├── score: Integer (1-5)
  └── content: String

工单状态日志(OrderLog)
  ├── orderId → RepairOrder
  ├── fromStatus/toStatus
  └── operatorId → User

公告(Notice)
  ├── title/content
  ├── adminId → User
  └── status: Integer (0下架/1发布)
```

**关系说明：**
- User(学生) 1:N RepairOrder（一个学生可提交多个工单）
- User(维修工) 1:N RepairOrder（一个维修工可承接多个工单）
- RepairOrder 1:1 Evaluation（一个已完成工单对应最多一条评价）
- RepairOrder 1:N OrderLog（工单每次状态变更产生一条日志）

### 2.4 分析结论

通过系统分析，确定以下关键点：

1. **系统边界**：平台对外提供 RESTful API，前端 Vue SPA 通过 HTTP + WebSocket 与后端交互；数据层使用 MySQL 持久化。

2. **核心对象**：RepairOrder 是系统的核心聚合根，贯穿所有业务流程；User 是多角色主体，其 role 字段驱动权限体系。

3. **核心业务规则**：工单状态转换严格受状态机约束（PENDING→ASSIGNED→PROCESSING→FINISHED→EVALUATED；任意状态→CANCELLED仅限学生本人或管理员）。

4. **分析支撑设计的依据**：领域模型直接映射为数据库表结构；状态机规则驱动 RepairOrderStateMachine 组件设计；多角色访问需求驱动 Spring Security + JWT + RBAC 架构选型。

## 3 系统设计

### 3.1 总体架构设计

系统采用**前后端分离**的两层部署架构：

```
┌─────────────────────────────────┐
│         前端（Vue 3 SPA）        │
│  Browser → Vite Dev Server      │
│  Vue Router / Pinia / Element+  │
└──────────────┬──────────────────┘
               │ HTTP(REST) / WebSocket
               ↓
┌─────────────────────────────────┐
│        后端（Spring Boot 3）     │
│  ┌──────────────────────────┐   │
│  │  Controller（表现层）    │   │
│  ├──────────────────────────┤   │
│  │  Service（业务逻辑层）   │   │
│  ├──────────────────────────┤   │
│  │  Mapper（数据访问层）    │   │
│  └──────────────────────────┘   │
│  Spring Security + JWT          │
│  MyBatis-Plus + MySQL           │
└─────────────────────────────────┘
```

**分层职责：**
- **Controller**：接收 HTTP 请求，参数校验，调用 Service，返回统一 Result 包装
- **Service**：业务逻辑，事务管理，状态机调用，WebSocket 通知触发
- **Mapper**：继承 MyBatis-Plus BaseMapper，提供 CRUD + 分页查询
- **Entity/VO/DTO**：数据承载；DTO 接收入参，VO 返回视图，Entity 映射数据库

### 3.4 关键交互设计

**时序图一：工单派单流程**

```
管理员Browser → OrderManageView → orderApi.assign() → POST /api/orders/assign
→ RepairOrderController.assign() → 获取 adminId
→ RepairOrderService.assignOrder()
  → StateMachine.assertTransit(PENDING→ASSIGNED) [失败抛 BusinessException]
  → RepairOrder.setWorkerId / setStatus(1) / setAssignTime
  → orderMapper.updateById()
  → writeLog(orderId, 0→1, adminId, "管理员派单")
  → pushNotify(workerId, {"type":"dispatch","message":"您有新的维修任务"})
    → NotifyWebSocketServer.sendToUser(workerId, json)
→ Result.success()
```

**时序图二：用户登录**

```
LoginView → authApi.login({username,password})
→ POST /api/auth/login → AuthController.login()
→ UserService.login()
  → AuthenticationManager.authenticate(token) [密码错误抛 BadCredentialsException]
  → userMapper.selectOne(username)
  → JwtTokenProvider.generateToken(username) → JWT字符串
→ LoginVO {token, UserVO}
→ authStore.setAuth(token, user) → localStorage
→ router.push('/')
```

### 3.5 质量属性设计

| 质量属性 | 设计决策 |
|---------|---------|
| **安全性** | JWT无状态认证；BCrypt密码加密；Spring Security方法级@PreAuthorize；JWT过滤器校验isEnabled拦截禁用用户；路由守卫前端角色校验 |
| **可修改性** | Controller/Service/Mapper三层分离，层间通过接口依赖；VO/DTO隔离视图与存储模型，修改数据库字段不影响接口 |
| **可测试性** | Service 层通过接口定义，便于 Mock 测试；GlobalExceptionHandler 统一异常处理，测试可断言错误码 |
| **性能** | MyBatis-Plus 分页插件避免全表扫描；关键字段建立索引；WebSocket 推送异步不阻塞主流程 |
| **可靠性** | 状态机保证工单状态转换合法性；@Transactional 保证工单状态变更与日志写入的原子性 |

### 4.2 方案比较与决策依据

**后端框架：Spring Boot vs Django vs Express**

| 维度 | Spring Boot | Django | Express |
|------|-------------|--------|---------|
| 语言 | Java | Python | Node.js |
| 安全生态 | Spring Security（成熟） | django-auth | 自行组合 |
| ORM | MyBatis-Plus/JPA | Django ORM | Sequelize |
| 团队熟悉度 | 高（专业课主讲） | 中 | 低 |
| 选择 | ✅ 选用 | — | — |

**选用 Spring Boot 理由**：团队最熟悉、安全框架最完整、与 Java 课程知识体系一致。

**前端框架：Vue 3 vs React vs Angular**

| 维度 | Vue 3 | React | Angular |
|------|-------|-------|---------|
| 学习曲线 | 低 | 中 | 高 |
| 模板语法 | HTML-like（直观） | JSX | 模板 |
| 状态管理 | Pinia（官方推荐） | Redux/Zustand | RxJS |
| 团队熟悉度 | 高 | 中 | 低 |
| 选择 | ✅ 选用 | — | — |

**数据库：MySQL vs PostgreSQL vs MongoDB**

选用 MySQL：数据结构固定（工单字段明确）、关系查询需求强（多表关联统计）、运维成本低、与 Spring Boot 生态集成最成熟。

**状态管理：Pinia vs Vuex**

选用 Pinia：Vuex 5 已废弃；Pinia 是 Vue 官方推荐的新一代状态管理库，Composition API 风格更自然，TypeScript 支持更完善。

### 5.2 核心功能实现

**（1）工单状态机**

工单状态共六种（0-5），转换规则通过 `OrderStatus.canTransitTo()` 枚举方法定义，`RepairOrderStateMachine.assertTransit()` 在每次状态变更前调用，不合法则抛出 `BusinessException`，确保工单不会出现跨状态跳转。

状态转换矩阵：

```
PENDING(0) → ASSIGNED(1) 或 CANCELLED(5)
ASSIGNED(1) → PROCESSING(2) 或 CANCELLED(5)
PROCESSING(2) → FINISHED(3) 或 CANCELLED(5)
FINISHED(3) → EVALUATED(4)
其他状态 → 不允许转换
```

**（2）JWT 认证链**

登录时 AuthenticationManager 验证密码（UserDetailsServiceImpl + BCryptPasswordEncoder），验证通过后 JwtTokenProvider 生成包含 username/userId/role 的 JWT。后续请求由 JwtAuthenticationFilter 解析 JWT、校验有效性和账号启用状态，将 UserDetails 写入 SecurityContext，@PreAuthorize 注解基于此进行方法级权限控制。

**（3）实时通知**

NotifyWebSocketServer 维护 `ConcurrentHashMap<Long, Session>`，key 为 userId。工单状态变更时 Service 层调用 `NotifyWebSocketServer.sendToUser(targetUserId, json)`，实时推送结构化消息（type/orderId/message）。WebSocket 连接建立时通过 query 参数传入 JWT 并校验，防止会话劫持。

**（4）分页查询**

所有列表接口均通过 MyBatis-Plus `Page<T>` + `LambdaQueryWrapper` 实现：

```java
Page<RepairOrder> pageResult = orderMapper.selectPage(
    new Page<>(page, size),
    new LambdaQueryWrapper<RepairOrder>()
        .eq(status != null, RepairOrder::getStatus, status)
        .orderByDesc(RepairOrder::getCreateTime)
);
return PageResult.of(pageResult.convert(this::toVO));
```

**（5）前端 RBAC 守卫**

路由配置中每条受保护路由带 `meta: { roles: [角色值] }`，路由守卫 `beforeEach` 读取 `authStore.user.role`，不在允许列表中则重定向到对应角色的默认页，防止越权访问。

### 5.3 关键代码分析

**代码段一：工单评价（双重幂等保护）**

```java
// RepairOrderServiceImpl.java
public void evaluate(Long studentId, EvaluationDTO dto) {
    RepairOrder order = orderMapper.selectById(dto.getOrderId());
    if (!studentId.equals(order.getStudentId()))
        throw new BusinessException("非本人工单");          // 所有权校验
    if (order.getStatus() != OrderStatus.FINISHED.code)
        throw new BusinessException("工单尚未完成");         // 状态前置校验
    Long count = evaluationMapper.selectCount(
        new LambdaQueryWrapper<Evaluation>()
            .eq(Evaluation::getOrderId, dto.getOrderId()));
    if (count > 0) throw new BusinessException("已评价过该工单"); // 幂等保护
    // ... 写入评价，更新状态
}
```

设计意图：三道防线（所有权→状态→幂等）确保评价操作的安全性，即使前端异常重复提交也不会产生脏数据。数据库 `uk_order_id` 唯一索引为最后一道兜底保证。

**代码段二：Axios 响应拦截器（统一错误处理）**

```typescript
// utils/http.ts
http.interceptors.response.use(
  (response) => {
    const res = response.data
    if (res.code !== 200) {
      ElMessage.error(res.message || '请求失败')
      return Promise.reject(new Error(res.message))
    }
    return res  // 直接返回 data 层，调用方无需 .data.data
  },
  (error) => {
    if (error.response?.status === 401) {
      authStore.logout(); router.push('/login')
    }
    // ...
  }
)
```

设计意图：将 HTTP 状态码处理和业务错误码处理集中在拦截器，所有 API 调用方只需处理成功场景，错误由拦截器统一弹窗提示。401 自动跳转登录页，避免页面出现空白状态。

**代码段三：路由角色守卫**

```typescript
// router/index.ts
router.beforeEach((to, _from, next) => {
  const authStore = useAuthStore()
  if (to.meta.requiresAuth && !authStore.isLoggedIn) { next('/login'); return }
  const roles = to.meta.roles as number[] | undefined
  if (roles && authStore.isLoggedIn) {
    const userRole = authStore.user?.role
    if (!roles.includes(userRole)) {
      const defaultPath = userRole === 0 ? '/student/orders'
        : userRole === 1 ? '/worker/orders' : '/'
      next(defaultPath); return
    }
  }
  next()
})
```

设计意图：前端 RBAC 是 UI 层防御，结合后端 `@PreAuthorize` 形成双层防护。即使用户直接在地址栏输入越权路由，也会被重定向到自己的默认页面。

**结论**：AI 是提升开发效率的强力工具，但不能取代开发者对业务逻辑的深入理解、对代码安全性的负责态度以及对整体架构的把控能力。正确的使用姿势是：**开发者主导架构和业务规则，AI 辅助实现和审查**。

## 9 总结与反思

### 9.1 技术收获

**后端方面：**
- 掌握了 Spring Security 6 的 JWT 无状态认证完整实现链路（从 UserDetailsService 到 JwtAuthenticationFilter 到 @PreAuthorize）
- 理解了 MyBatis-Plus 的 LambdaQueryWrapper 条件构造器和分页插件的正确用法
- 实践了枚举驱动的状态机设计，理解了"用代码表达业务规则"而非硬编码条件判断的重要性

**前端方面：**
- 熟悉了 Vue 3 Composition API 的响应式编程模式（ref/reactive/computed）
- 掌握了 Pinia store 的设计方法，理解了前端状态与 UI 的解耦
- 理解了 Axios 拦截器在统一处理认证和错误方面的价值

**工程方面：**
- 实践了前后端分离架构，理解了 Vite proxy 的开发代理原理
- 体验了 AI 辅助代码审查的完整流程，理解了多角度 Review 的价值

### 9.2 工程收获

- **分层设计的价值**：Controller 只做 HTTP 入参/出参转换，业务逻辑集中在 Service，当需求变化（如取消工单新增所有权校验）时，只需修改 Service 层，Controller 改动最小化。

- **接口先行的重要性**：Service 先定义接口，后写实现类，使得前端可以根据接口文档并行开发，减少了联调阶段的摩擦。

- **安全是设计的一部分**：本项目的安全问题大多源于"功能先行、安全后补"的开发顺序。在设计阶段就应该将认证、授权、输入校验纳入考虑，而不是等功能完成后再补丁式修复。

### 9.3 不足与改进方向

| 不足 | 改进方向 |
|------|---------|
| Redis 配置但未实际使用 | 将 JWT 黑名单（注销令牌）、短信验证码等放入 Redis 实现 |
| 无单元测试和集成测试 | 补充 Service 层单元测试（JUnit 5 + Mockito）和接口集成测试 |
| 图片上传功能缺失 | 实现报修图片上传，对接 OSS 或本地文件服务 |
| 统计图表数据为模拟数据 | 实现按日期聚合的真实趋势数据接口 |
| 维修工技能匹配 | 派单时根据工单类别和维修工 t_worker_skill 自动推荐维修工 |
| 无移动端适配 | 使用 uni-app 开发微信小程序端，支持学生手机报修 |

### 9.4 课程设计体会

本次课程设计完成的不只是"一个能运行的系统"，而是尝试经历了从需求分析、领域建模、架构设计到编码实现、测试审查的完整工程化软件开发过程。

在 AI 辅助开发高度普及的今天，代码生成的门槛大幅降低，但这反而让工程师的**架构判断力**和**安全意识**变得更加重要。AI 能快速生成实现代码，但它不了解系统边界、不清楚业务语义、不负责安全责任。本项目中，WebSocket 会话劫持漏洞和 JWT 禁用绕过漏洞都是 AI 在初始生成时未考虑到的，需要开发者主动识别和修复。

这让我理解了软件工程专业所强调的"工程化能力"的核心：在复杂度和不确定性中做出正确决策，对系统的安全性、可维护性、可扩展性负责——这是任何工具都无法替代的专业判断力。
